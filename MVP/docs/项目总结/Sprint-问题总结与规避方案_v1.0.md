## 一、概述

本文档总结 MVP 新闻爬虫与 AI 整理系统开发过程中遇到的问题、解决方案，以及未来项目的规避建议。

## 二、问题清单与解决方案

### 2.1 爬虫相关问题

#### 问题 1：radarai.top React SPA 无法正确抓取
**问题描述**：
- radarai.top 是 React 单页应用，使用 JavaScript 动态渲染内容
- 直接抓取 HTML 无法获取新闻列表

**解决方案**：
- 分析发现页面有 SSR（服务端渲染）内容
- 从 `div#ssr-article-list` 中提取 JSON 数据
- 使用 SSR 数据解析新闻列表

**规避建议**：
- 对于现代前端框架网站，先检查是否有 SSR 内容
- 查找 `window.__INITIAL_STATE__`、`__NEXT_DATA__` 等全局变量
- 检查 JSON-LD 等结构化数据

---

#### 问题 2：jqzx 网站 Target div not found
**问题描述**：
- 机器之心网站改版，原有选择器失效
- 只抓取到 1 条新闻

**解决方案**：
- 评估后决定完全移除 jqzx 爬虫
- 从 `CRAWLER_REGISTRY` 中删除

**规避建议**：
- 定期检查爬虫是否正常工作
- 建立爬虫监控告警机制
- 为每个爬虫设计多个备用选择器

---

### 2.2 配置相关问题

#### 问题 3：YAML 时间格式解析异常
**问题描述**：
- `11:30` 格式在 YAML 中被解析为整数（分钟数）
- 690 = 11×60 + 30
- 导致 Pydantic 验证失败（期望 List[str]，得到 List[int]）

**解决方案**：
1. 保存时用单引号括起来：`'11:30'`
2. 读取时检测整数并转换回 "HH:MM" 格式
3. 手动构建 YAML 确保 cron_times 正确引用

**规避建议**：
- 时间、日期等特殊格式在 YAML 中必须加引号
- 配置文件变更后立即验证
- 编写配置读写的单元测试

---

#### 问题 4：配置修改后 Scheduler 不生效
**问题描述**：
- Web 界面修改配置后，Scheduler 仍使用旧配置
- Scheduler 启动时只加载一次配置

**解决方案**：
- 保存配置后自动重启 Scheduler
- 给保存按钮添加 loading 效果

**规避建议**：
- 设计配置热重载机制
- 或者明确提示用户需要重启服务

---

### 2.3 LLM 相关问题

#### 问题 5：LLM 处理超时
**问题描述**：
- API 响应时间不稳定，偶尔超时
- 超时后整个流程失败

**解决方案**：
- 实现关键词评分降级方案
- LLM 失败时自动使用关键词筛选

**规避建议**：
- 设置合理的超时时间
- 始终提供降级方案
- 记录 LLM 成功率指标

---

#### 问题 6：LLM 排序标识不正确
**问题描述**：
- 日志显示 LLM 处理完成，但通知显示"关键词筛选"
- Scheduler 初始化 LLM 时未传递 `use_llm_for_ranking` 参数

**解决方案**：
- Scheduler 初始化时传递所有 LLM 配置参数

**规避建议**：
- 参数传递后立即验证
- 编写初始化逻辑的单元测试

---

### 2.4 Web 界面相关问题

#### 问题 7：Flask 403 Forbidden 错误
**问题描述**：
- template_folder 和 static_folder 路径配置错误
- 使用相对路径导致找不到模板文件

**解决方案**：
- 使用 `os.path.dirname(os.path.abspath(__file__))` 构建绝对路径

**规避建议**：
- Web 框架始终使用绝对路径
- 开发环境先测试路径配置

---

#### 问题 8：Python 模块导入错误
**问题描述**：
- `python -m src/news_agent/main.py` 报 ModuleNotFoundError
- 包结构和执行方式不匹配

**解决方案**：
- 直接执行脚本或正确设置 PYTHONPATH

**规避建议**：
- 提供正确的启动脚本
- 在文档中明确启动方式

---

### 2.5 数据库相关问题

#### 问题 9：需记录已发送新闻避免重复推送
**问题描述**：
- 每次爬虫都发送相同新闻
- 需要追踪哪些新闻已发送过

**解决方案**：
- 在 NewsModel 中添加 `sent_at` 字段
- 添加数据库迁移逻辑
- Repository 增加 `mark_as_sent()`、`get_unsent()`、`is_sent()` 方法
- 所有新闻都已发送时，发送"无新新闻"通知

**规避建议**：
- 设计初期考虑去重和状态追踪
- 数据库变更考虑迁移方案

---

### 2.6 Git 相关问题

#### 问题 10：初始提交包含 .DS_Store
**问题描述**：
- macOS 自动生成的 .DS_Store 被提交
- 影响仓库整洁

**解决方案**：
- 完善 .gitignore
- 从 git 缓存中删除

**规避建议**：
- 项目开始前先配置好 .gitignore
- 使用标准的 .gitignore 模板

---

## 三、未来项目规避建议

### 3.1 开发流程
1. **先设计后编码**：复杂功能先写技术设计文档
2. **测试先行**：核心逻辑先写单元测试
3. **频繁验证**：每个模块完成后立即测试
4. **代码审查**：重要变更需要 review

### 3.2 配置管理
1. **配置验证**：添加配置 schema 验证
2. **默认值**：所有配置项提供合理默认值
3. **环境变量**：敏感配置使用环境变量
4. **配置热重载**：考虑使用 etcd、Consul 等配置中心

### 3.3 爬虫开发
1. **多选择器**：每个网站设计 2-3 个选择器
2. **监控告警**：爬虫失败及时通知
3. **爬虫池**：设计可插拔的爬虫架构
4. **异常恢复**：单次失败不影响整体流程

### 3.4 数据库设计
1. **状态字段**：考虑添加 created_at、updated_at、status 等字段
2. **索引设计**：为常用查询字段添加索引
3. **迁移方案**：预留数据迁移机制
4. **备份策略**：定期备份重要数据

### 3.5 LLM 集成
1. **降级方案**：LLM 必须有降级逻辑
2. **超时重试**：合理的重试策略
3. **成本控制**：监控 token 使用量
4. **结果缓存**：相同输入缓存结果

### 3.6 文档记录
1. **问题记录**：遇到问题立即记录
2. **变更记录**：重要变更写清楚原因
3. **决策记录**：技术选型记录决策过程
4. **运维手册**：编写部署、监控、故障排查手册

## 四、总结

本项目在开发过程中遇到了爬虫、配置、LLM、Web 界面等多方面问题，通过快速迭代和持续优化最终解决。这些经验和教训对未来项目有重要参考价值。

**核心收获**：
1. 设计比编码更重要
2. 测试是质量的保障
3. 文档是团队协作的基础
4. 问题记录是宝贵的知识资产
