# Claude 开发规则

## Agent 项目群开发规范

### 规则 1：代码完成后必须添加对应单元测试用例
- **适用范围**：所有核心业务逻辑代码
- **测试覆盖要求**：
  - 正常路径
  - 边界条件
  - 错误/异常路径

### 规则 2：编写代码后必须编译确保可以通过编译检测

### 规则 3：单元测试用例必须可以通过测试

### 规则 4：代码修改需要明确指示
- **修改代码前**：必须等待用户的明确指示才能执行代码修改
- **有效指令**：包含"执行"、"优化"、"修复"、"解决"、"写"、"添加"、"删除"、"更新"等明确要求修改代码的指令
- **禁止行为**：在没有明确指令的情况下，不得主动修改代码文件
- **文档任务例外**：文档更新、文件整理、需求分析等非代码修改任务可以正常进行

### 规则 5：先检查再动手，设计规划先行
- **核心理念**：任何任务都应该先检查现状、理解需求，再动手执行。不仅适用于代码设计，也适用于产品 PRD、项目管理、视觉设计等所有类型的任务。
- **先检查再动手**：
  - 执行任务前，先检查代码库/文档库中是否已有相关实现
  - 理解现有实现的设计思路和当前状态
  - 区分"需新增实现" vs "需验证确认" vs "已完成无需操作"
  - 避免重复劳动，保持代码/文档一致性
- **技术规划设计**（代码类任务）：
  - 修改/新增功能前，先在 `docs/技术设计/` 目录下完成技术方案设计
  - 设计内容包括：接口定义、数据流、架构选型、代码示例等
  - 重大变更需要评审后再开始编码
- **单元测试设计**（代码类任务）：
  - 编码前先设计单元测试用例大纲
  - 明确测试覆盖范围：正常路径、边界条件、异常路径
  - 测试设计与技术设计同步完成
- **例外情况**：
  - 简单 bug 修复（单行或少量代码修改）可简化流程
  - 文档更新、代码格式调整等非功能修改不需要技术设计

### 规则 6：Sprint 任务结束后必须输出必要的交付物文档
- **适用范围**：每个 Sprint 任务完成后
- **必须输出的文档**：
  - SprintXX-交付物清单_v1.0.md：列出本 Sprint 所有交付物
  - SprintXX-测试报告_v1.0.md：包含功能测试、单元测试结果
  - SprintXX-开发总结与问题反馈_v1.0.md：Sprint 回顾、技术要点、经验教训
- **文档存放位置**：`docs/{项目版本}/项目管理/sprintXX/`
- **可选文档**（根据 Sprint 内容）：
  - SprintXX-真机测试报告_v1.0.md：如有真机测试需求
  - 技术设计文档：如有新技术方案
- **文档要求**：
  - 参考过往 Sprint 文档格式
  - 内容真实反映 Sprint 执行情况
  - 包含完成度统计和验收结论

### 规则 7：所有输出材料必须保证数据准确性
- **适用范围**：所有输出的文档、代码注释、任务说明等材料
- **数据准确性要求**：
  - 所有数据必须有可靠来源或真实记录
  - 禁止编造、臆测或复制不真实的数据
  - 引用过往数据时必须核对来源
- **无把握时的处理方式**：
  - 如对数据有疑问或无把握，必须添加 **（待确认）** 标识
  - 对于预测性、估算性数据，明确标注为"预计"、"估算"
  - 对于需要后续补充的数据，标注"待记录"、"待填写"

### 规则 8：所有输出/优化/补充材料必须包含变更记录用于追溯
- **适用范围**：所有输出的文档、代码变更说明、交付物材料
- **变更记录要求**：
  - 所有文档必须在开头包含变更记录表格
  - 变更记录必须包含：版本、日期（精确到分钟）、修改人、修改说明
  - 每次修改文档都必须追加一条变更记录
- **变更记录格式**：
  ```markdown
  ## 文档变更记录

  | 版本 | 日期 | 修改人 | 修改说明 |
  |-----|------|--------|---------|
  | v1.0 | 2026-04-13 10:15 | {你的名字} | 初始版本 |
  ```
- **代码注释要求**：
  - 复杂代码修改应在注释中包含修改时间和修改人
  - 示例：`# Modified by Claude at 2026-04-13 14:30: 修复 XX 问题`

### 规则 9：Sprint 提交规范（使用 git 时间作为客观工时记录）
- **核心理念**：使用 git commit 时间作为客观的开发耗时记录，替代主观的工时填写
- **Sprint 开始时必须提交**：
  - 现状分析文档
  - 需求理解文档
  - 任务拆分计划
  - 技术设计文档（如需要）
- **Sprint 结束时必须提交**：
  - 测试报告
  - 交付物清单
  - 开发总结与问题反馈
- **文档要求**：
  - 以上文档如缺失且必要均应补充并提交
  - 所有文档遵循规则 8 的变更记录要求
- **git 提交规范**：
  - 每次重要变更都应独立 commit
  - Commit message 清晰说明变更内容
  - 通过 git log 可完整追溯 Sprint 开发过程

## 项目概述

本项目是一个 Python Agent 项目群，包含多个独立的 Agent 服务，协同工作完成复杂任务。

### 当前项目

- **MVP**：新闻爬虫与 AI 整理系统
  - 功能：多网站新闻爬虫 + 大模型智能整理 + 飞书/邮件通知
  - 文档位置：`MVP/docs/需求文档/PRD-MVP-新闻爬虫与AI整理系统_v1.0.md`

## Agent 开发规范

### 每个 Agent 应遵循的原则

1. **单一职责**：每个 Agent 专注于一类任务
2. **可配置**：通过配置文件或环境变量调整行为
3. **可测试**：核心逻辑独立，易于单元测试
4. **日志完善**：关键操作留有日志记录
5. **错误处理**：合理的异常捕获和重试机制

### Agent 目录结构模板


## Git 提交规范

提交信息格式：
```
<type>(<agent>): <description>

- 详细说明 1
- 详细说明 2
```

Type 类型：
- `feat`：新功能
- `fix`：修复 bug
- `docs`：文档更新
- `test`：测试相关
- `refactor`：重构
- `style`：代码格式
- `chore`：构建/工具相关

示例：
```
feat(crawler): 新增36kr网站爬虫支持

- 实现36kr新闻列表解析
- 添加发布时间和来源提取
- 增加单元测试覆盖
```

## 注意事项

1. **文档与代码分离**：每个 Agent 有自己的 `docs/` 目录存放专属文档
2. **单元测试**：核心逻辑必须有对应的单元测试，测试放在 `src/tests/` 目录
3. **测试通过**：确保所有单元测试通过
4. **虚拟环境**：每个 Agent 推荐使用独立的虚拟环境
5. **依赖管理**：使用 requirements.txt 或 pyproject.toml 管理依赖
