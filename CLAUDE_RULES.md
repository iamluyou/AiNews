# CLAUDE 开发规则 - 强制检查清单

> ⚠️ 重要：每次新 Session 开始、每次开始编码前，必须先读此文件！

## 开始任务前必须做的检查

### 1. 先检查现状
- [ ] 我是否已经搜索和阅读了相关的现有代码？
- [ ] 我是否理解了当前的实现方式？
- [ ] 这个功能是否已经实现了？

### 2. 设计规划先行
- [ ] 这个变更是否需要写技术设计文档？
  - 新增功能 → 需要
  - 多文件修改 → 需要
  - 架构变更 → 需要
  - 简单 bug 修复 → 不需要
- [ ] 如果需要设计，我是否先写了 `docs/技术设计/` 下的设计文档？

### 3. 单元测试
- [ ] 核心业务逻辑是否有对应的单元测试？
- [ ] 测试是否覆盖了：正常路径 / 边界条件 / 异常路径？

### 4. 代码编译/测试通过
- [ ] 代码写完后是否编译/运行检查过？
- [ ] 单元测试是否全部通过？

### 5. 代码修改需要明确指示
- [ ] 用户是否给出了明确的修改指令？
  - "执行"、"优化"、"修复"、"解决"、"写"、"添加"、"删除"、"更新" → 可以修改
  - 讨论、分析、建议 → 不能修改代码

## CLAUDE.md 核心规则速查

| 规则 | 内容 | 什么时候必须遵守 |
|------|------|-----------------|
| 规则 1 | 核心业务逻辑必须写单元测试 | 所有核心业务代码 |
| 规则 2 | 代码必须编译通过 | 所有代码修改 |
| 规则 3 | 单元测试必须通过 | 所有代码修改 |
| 规则 4 | 代码修改需要明确指示 | 任何代码修改前 |
| 规则 5 | 先检查再动手，设计规划先行 | 复杂功能、新增功能 |

## 本项目特定规则

### MVP 项目额外约定
- 修改配置后必须验证：配置能否被正确加载？
- 修改 scheduler 后必须验证：与 run_once.py 逻辑是否一致？
- 数据库变更必须有迁移逻辑

### 禁止行为
- ❌ 不要在没有明确指令时主动修改代码
- ❌ 不要为了"优化"而修改不在任务范围内的代码
- ❌ 不要忘记写单元测试

## 每次编码前的自我确认

在开始写代码前，先在心里问自己：

1. "我读了 CLAUDE_RULES.md 了吗？"
2. "我需要先写技术设计吗？"
3. "我会记得写单元测试吗？"
4. "用户明确让我修改代码了吗？"

如果以上任何一个答案是"No"，先停下来解决。
